# ТЗ: NFC Quest — веб-приложение для NFC-квеста

## 1. Обзор

Веб-приложение для проведения NFC-квеста на мероприятии. Участники сканируют NFC-метки смартфоном, получают или теряют баллы, соревнуются за место в рейтинге. Победитель определяется по наибольшему количеству баллов к моменту окончания квеста.

---

## 2. Механика игры

### 2.1 Участники
- При первом сканировании **любой** NFC-метки фронтенд проверяет наличие UUID участника в localStorage.
- Если UUID отсутствует — показывается форма регистрации: вводится никнейм, генерируется UUID, данные отправляются на сервер.
- В дальнейшем каждый запрос к серверу содержит UUID участника и ID метки.
- Привязка к устройству через localStorage — никакой авторизации по паролю нет.

### 2.2 Метки

Все метки технически одинаковы — разница только в назначенной стратегии и параметрах. Физически разделяются на два вида (условно):

| Вид (условный) | Описание |
|----------------|----------|
| **Метка участника** | Выдаётся участнику на старте. Обычно стратегия `one_time_global` с `+10` — сканирование даёт баллы **сканирующему**. |
| **Спрятанная метка** | Физически спрятана в помещении. Может иметь любую стратегию и любое значение баллов (включая отрицательное). |

С точки зрения бэкенда — это просто метки с разными стратегиями. Никакого отдельного поля `type` нет.

### 2.3 Стратегии начисления баллов (модуль `ScoringStrategy`)
Каждой метке при создании назначается одна стратегия. Стратегия — это функция/класс на бэкенде, которая получает `(tag_id, player_id)` и возвращает `delta_points` (может быть отрицательным).

Обязательные стратегии:

| Стратегия | Поведение |
|-----------|-----------|
| `one_time_global` | Метка срабатывает один раз для **любого** участника, после чего блокируется навсегда. |
| `one_time_per_player` | Метка срабатывает один раз для каждого участника (другой участник может использовать). |
| `unlimited` | Метка всегда возвращает фиксированное значение, без ограничений. |
| `random` | Возвращает случайное значение из заданного диапазона (min, max), может быть отрицательным. |

Стратегии реализуются как подключаемые модули — добавление новой стратегии не требует изменения основного кода.

## 3. Функциональные требования

### 3.1 Фронтенд

#### Страница сканирования (`/tag/<tag_id>`)
- Открывается по URL из NFC-метки (вида `https://example.com/tag/ABC123`).
- Если участник не зарегистрирован (нет UUID в localStorage) — показывает форму с полем «Никнейм», кнопкой «Начать игру». После отправки вызывается `POST /api/register`, UUID сохраняется в localStorage, затем сразу выполняется `POST /api/scan` для текущей метки.
- Если участник уже зарегистрирован — сразу отправляет запрос `POST /api/scan` с `{tag_id, player_id}`.
- Показывает результат сканирования: `+N баллов` / `-N баллов` / `уже использована` / `квест не начался` / `квест завершён`.

#### Страница табло (`/scoreboard`)
- Показывает таблицу участников: место, никнейм, баллы.
- Таймер состояния игры — отображается в зависимости от статуса:
  - `not_started`: обратный отсчёт до времени начала (`starts_at`)
  - `active`: обратный отсчёт до времени окончания (`ends_at`)
  - `finished`: сообщение «Награждение в ХХХХ» (текст из поля `award_message` в настройках игры)
- Обновление в реальном времени через **WebSocket** (`flask-socketio`). Сервер пушит обновлённый рейтинг всем подключённым клиентам при каждом успешном сканировании.
- Подходит для отображения на большом экране в холле.
- При подключении клиент сразу получает текущее состояние (рейтинг + статус игры + `starts_at`/`ends_at`/`award_message`).

### 3.2 Бэкенд (Flask, Python)

#### Endpoints

| Метод | URL | Описание |
|-------|-----|----------|
| `POST` | `/api/register` | Регистрация нового участника |
| `POST` | `/api/scan` | Обработка сканирования метки |
| `GET` | `/api/scoreboard` | Получение текущего рейтинга |
| `GET` | `/api/game_status` | Статус игры (not_started / active / finished), время окончания |

#### Логика `POST /api/register`
Вызывается однократно при первом сканировании, если участник ещё не зарегистрирован.
1. Принять `{player_id, nick}`.
2. Проверить уникальность `player_id` и ника.
3. Сохранить участника в БД с нулевым счётом.
4. Вернуть `{player_id, nick, points: 0}`.

После успешной регистрации фронтенд немедленно делает запрос `POST /api/scan` с той меткой, которая спровоцировала регистрацию.

#### Логика `POST /api/scan`
Принимает `{tag_id, player_id}`. Участник должен быть уже зарегистрирован.
1. **Rate limiting**: проверить, что с момента последнего сканирования данным `player_id` прошло не менее 1 секунды. Если нет — вернуть `HTTP 429 Too Many Requests`.
2. Проверить статус игры (если не активна — вернуть ошибку).
3. Найти участника по `player_id`. Если не найден — вернуть ошибку (фронт должен был сначала вызвать `/api/register`).
4. Найти метку по `tag_id`. Если метка неизвестна — вернуть ошибку.
5. Применить стратегию метки: получить `delta_points`.
6. Начислить/списать баллы участнику.
7. Обновить время последнего сканирования для `player_id`.
8. Вернуть `{delta: N, total: M, message: "..."}`.

Rate limit реализуется на бэкенде через in-memory хранилище (`dict` с `player_id → last_scan_time`). Сброс при перезапуске сервера допустим.

### 3.3 Админ-панель (`/admin`)

**Авторизация:** все маршруты `/admin/*` защищены паролем. Неавторизованный запрос редиректит на `/admin/login`. Вход — через форму с паролем, сессия хранится в cookie (Flask session, `SECRET_KEY` из `.env`). Пароль задаётся через `ADMIN_PASSWORD` в `.env`.

#### Раздел: Управление игрой
Настройки игры (сохраняются кнопкой «Сохранить»):
- Время начала (`starts_at`) — абсолютное datetime.
- Время окончания (`ends_at`) — абсолютное datetime.
- Сообщение о награждении (`award_message`) — текст, показываемый на табло после окончания игры (например: «Награждение в 18:00, зал А»).

Управляющие кнопки (каждая требует подтверждения):
- **«Запустить игру»** — принудительный перевод в статус `active` прямо сейчас (не дожидаясь `starts_at`). Обновляет `starts_at` на текущее время.
- **«Остановить игру»** — принудительное завершение (`active` → `finished`). Обновляет `ends_at` на текущее время.
- **«Очистить список участников»** — удалить всех участников и их баллы.
- **«Очистить список меток»** — удалить все метки.

Статус игры определяется автоматически по времени: если текущее время между `starts_at` и `ends_at` — игра `active`; до `starts_at` — `not_started`; после `ends_at` — `finished`. Принудительные кнопки перезаписывают `starts_at`/`ends_at`.

#### Раздел: Управление метками
- **Пакетное создание**: форма — выбрать стратегию, задать параметры (баллы / диапазон), указать количество штук (`N`). Сервер генерирует `N` меток с автоматическими TAG_ID.
- **Результат создания**: список сгенерированных URL (`https://<BASE_URL>/tag/<TAG_ID>`) — для копирования и последующей записи в NFC-метки.
- **Список меток**: таблица со всеми метками, столбцы: TAG_ID, стратегия, параметры, кол-во сканирований (всего и по уникальным участникам), статус (активна / заблокирована).
- **Действия над меткой**: удалить метку; сбросить счётчик использований (разблокировать `one_time_global`).

#### Раздел: Участники
- Таблица участников: ник, UUID, баллы, время регистрации.
- Действия: удалить участника, скорректировать баллы вручную (без указания причины).

#### Раздел: Лог событий
- Хронологический лог всех сканирований: время, ник участника, TAG_ID метки, стратегия метки, начислено баллов, итоговый счёт участника после события.
- Фильтрация по участнику и по метке.

## 4. Модель данных

### `Player`
| Поле | Тип | Описание |
|------|-----|----------|
| `id` | UUID | Уникальный идентификатор |
| `nick` | string | Никнейм |
| `points` | int | Текущий счёт |
| `registered_at` | datetime | Время регистрации |

### `Tag`
| Поле | Тип | Описание |
|------|-----|----------|
| `id` | string | ID метки (записан в NFC), генерируется сервером |
| `label` | string / null | Человекочитаемое название метки (опционально, для удобства в админке) |
| `strategy` | string | Название стратегии |
| `strategy_params` | JSON | Параметры стратегии (баллы, диапазон и т.д.) |
| `is_blocked` | bool | Заблокирована ли метка глобально (для `one_time_global`) |
| `created_at` | datetime | Время создания |

### `ScanEvent`
| Поле | Тип | Описание |
|------|-----|----------|
| `id` | int | PK |
| `tag_id` | string | ID метки |
| `player_id` | UUID | Кто сканировал |
| `delta_points` | int | Начислено/списано |
| `scanned_at` | datetime | Время |

### `GameSettings`
Единственная запись в таблице (сущность игры одна).

| Поле | Тип | Описание |
|------|-----|----------|
| `starts_at` | datetime / null | Запланированное время начала |
| `ends_at` | datetime / null | Запланированное время окончания |
| `award_message` | string | Сообщение о награждении, показываемое после окончания |

Статус (`not_started` / `active` / `finished`) — вычисляется на бэкенде по текущему времени и полям `starts_at` / `ends_at`, не хранится отдельно.

## 6. Деплой

- `docker-compose.yml` поднимает один контейнер с Flask.
- База данных — SQLite-файл, монтируется как volume для персистентности.
- Конфигурация через переменные окружения (`.env`): `ADMIN_PASSWORD`, `SECRET_KEY`, `BASE_URL` (для генерации URL меток).
- Порт 5000 по умолчанию.

---

## 7. URL-схема NFC-меток

Каждая метка хранит URL вида:
```
https://<BASE_URL>/tag/<TAG_ID>
```

`TAG_ID` — произвольная строка (например, `A1B2C3` или UUID). Генерируется и вписывается в метку при регистрации через админку.

**URL:** `/tag/<tag_id>` (участник уже зарегистрирован, после сканирования)

**Контекст:** участник только что отсканировал метку, сервер вернул результат

**Состояния (один и тот же экран, разные варианты отображения):**

| Состояние | Визуал |
|-----------|--------|

**Постоянные элементы:** ник участника, текущие баллы вверху (или внизу).

## 9. Ограничения и допущения

- Один участник — одно устройство (без механизма восстановления аккаунта по другому устройству).
- Нет защиты от подделки UUID на клиенте (квест — развлекательный, не требует строгой защиты).
- Перезапись NFC-меток участниками физически возможна, но не предусмотрена в приложении — сервер просто не найдёт перезаписанный ID, если он не зарегистрирован в базе.
- Отрицательный баланс участника допускается.
- Интернет на площадке предполагается.
